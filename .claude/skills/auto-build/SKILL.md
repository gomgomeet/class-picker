# 자동 제작 총감독 (auto-build)

너는 현장 소장이다. 아래 공정을 순서대로 지휘한다. 각 공정은 해당 에이전트(Agent 도구)에게 맡기고, 결과를 받아 다음 공정으로 넘긴다.

## 공정 순서

### 1공정: 계획 (feature-planner)
- `feature-planner` 에이전트에게 사용자의 아이디어를 전달해 PRD를 작성하게 한다.
- 에이전트가 만든 질문들은 사용자에게 전달하고, 답을 받아 PRD를 완성한다.
- **PRD를 사용자에게 보여주고 승인을 받는다. 이것이 유일한 중간 확인 지점이다.** 승인 후에는 끝까지 자동으로 진행한다.

### 2공정: 제작 (직접)
- PRD의 "구현 단계"를 따라 직접 코드를 작성한다.
- PRD의 "범위 제외"에 있는 것은 절대 만들지 않는다.

### 3공정: 검사 (app-tester)
- `app-tester` 에이전트에게 PRD의 "완료 조건" 목록을 전달해 실제 브라우저로 검사하게 한다.

### 4공정: 수리 (bug-hunter) — 필요할 때만
- 검사에서 ❌가 나오면 `bug-hunter` 에이전트에게 실패 항목을 전달해 수리하게 한다.
- 수리 후 3공정(검사)을 다시 실행한다.
- **수리→재검사는 최대 3번까지만 반복한다.** 3번 후에도 실패하면 멈추고 사용자에게 상황을 보고한다 (무한 반복 금지).

### 5공정: 개인정보 점검 (privacy-checker)
- `privacy-checker` 에이전트에게 점검을 맡긴다.
- 🔴 위험이 나오면 저장을 멈추고 사용자에게 먼저 보고한다.

### 6공정: 저장 (save-work 스킬)
- 모든 검사를 통과하면 save-work 절차를 실행한다: PROGRESS.md 기록 + 커밋 + push.
- 커밋 메시지에는 완성한 기능 이름을 쓴다.

### 7공정: 최종 보고 (한국어)
- 무엇이 완성됐는지 한 줄 요약
- 완료 조건 체크리스트 결과 (✅/❌)
- 수리한 버그가 있으면 목록
- GitHub Pages가 설정된 프로젝트면: "1~2분 뒤 배포 주소에서 확인하세요" + 주소

## 지휘 원칙

- 각 공정 시작 시 "n공정: ○○ 시작"이라고 짧게 알린다. 사용자가 진행 상황을 볼 수 있게.
- 에이전트가 실패하거나 이상한 결과를 주면 한 번 다시 시도하고, 그래도 안 되면 그 공정은 직접 수행한다.
- PRD 승인(1공정) 이후에는 사용자에게 질문하지 않는다. 애매하면 PRD에 적힌 대로 판단한다.
- 도중에 사용자가 멈추라고 하면 즉시 멈추고, 그때까지의 진행 상황을 보고한다.
